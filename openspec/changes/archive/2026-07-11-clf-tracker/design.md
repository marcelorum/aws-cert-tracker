# Design: AWS CLF-C02 Certification Tracker

## Technical Approach

React 19 + Vite 6 SPA with Dexie.js (IndexedDB) for persisted data and Zustand for ephemeral UI state. Components are organized by domain feature (checklist, progress, resources) with a shared layout shell. Seed data is a static TypeScript file imported at build time and inserted on first-visit detection. Reactive rendering via Dexie `liveQuery` wrapped in React hooks — no manual refresh.

This maps to the proposal's **Approach B (Lightweight SPA)** and satisfies all 4 specs (topic-checklist, progress-tracking, local-persistence, resource-management).

## Architecture Decisions

### Decision: Persistence boundary

| Option | Tradeoff | Decision |
|--------|----------|----------|
| State-only in Zustand (no DB) | Lost on reload, violates `local-persistence` spec | ❌ Rejected |
| Dexie for data + Zustand for UI | Clear separation: persisted vs. ephemeral. Live queries auto-subscribe | ✅ **Chosen** |
| Single Zustand store with localStorage persist | Mixes concerns, no query/index capabilities, manual serialization | ❌ Rejected |

### Decision: Data reactivation pattern

| Option | Tradeoff | Decision |
|--------|----------|----------|
| `liveQuery` in components directly | Duplicate subscription logic across files | ❌ Rejected |
| `useLiveQuery()` in custom hooks per table | Single-responsibility hooks. Components call `useTopics(domainId)` | ✅ **Chosen** |
| Zustand middleware wrapping Dexie | Adds unnecessary indirection, fights reactive nature of liveQuery | ❌ Rejected |

### Decision: Tree navigation state

| Option | Tradeoff | Decision |
|--------|----------|----------|
| Derive collapsed/expanded from query params | URL encodes nav state, works with browser back | ❌ Over-engineered for SPA |
| Zustand `expandedDomains` + `expandedTopics` | Simple Sets, persisted to sessionStorage only (not IndexedDB) | ✅ **Chosen** |
| React context | Prop-drilling avoided, but no persistence and re-renders on every toggle | ❌ Rejected |

### Decision: Seed data delivery

| Option | Tradeoff | Decision |
|--------|----------|----------|
| Fetch from CDN on first load | Breaks offline constraint | ❌ Rejected |
| Inline JSON in a `.ts` file | Build-time import, tree-shakeable, no runtime fetch | ✅ **Chosen** |
| Service worker inject | Complex, hard to version | ❌ Rejected |

## Data Flow

```
                    ┌──────────────────────────┐
                    │      seed-data.ts         │
                    │  (build-time import)      │
                    └────────────┬─────────────┘
                                 │ first visit check
                                 ▼
┌──────────────┐   liveQuery    ┌──────────────────┐   Zustand   ┌────────────────┐
│   Dexie DB   │◄──────────────►│   Custom Hooks    │◄──────────►│  React Pages   │
│  CLFTrackerDB│                │  useTopics()      │            │  + Components  │
│              │   CRUD ops     │  useDomains()     │   reads    │                │
│  exam        │◄───────────────│  useResources()   │            │  Dashboard     │
│  domain      │                │  useProgress()    │            │  Checklist     │
│  topic       │  tx write      │                   │            │  TopicDetail   │
│  resource    │◄───────────────│  Store actions    │──────────► │                │
└──────────────┘                └──────────────────┘   writes   └────────────────┘
                                                                    │
                                              ┌─────────────────┐   │
                                              │  Zustand UIStore │◄──┘
                                              │  sidebarOpen     │
                                              │  expandedDoms    │
                                              │  activeTopicId   │
                                              └─────────────────┘
```

Status update path: click badge → `topicTable.update(id, {status})` → liveQuery emits → React re-renders → progress recalculates — no manual dispatch.

## Dexie Schema

```typescript
// src/db/schema.ts
const db = new Dexie('CLFTrackerDB');

db.version(1).stores({
  exam:     '++id, name',
  domain:   '++id, examId, weight',
  topic:    '++id, domainId, parentTopicId, status',
  resource: '++id, topicId, resourceType, url, createdAt',
});
```

Relationships: Exam 1→N Domain, Domain 1→N Topic, Topic 1→N Resource. Sub-topics reference `topic.parentTopicId` (self-referencing FK).

## Interfaces / Contracts

```typescript
// src/lib/types.ts
type TopicStatus = 'not_started' | 'in_progress' | 'completed';
type ResourceType = 'video' | 'article' | 'practice_exam' | 'aws_doc' | 'note' | 'flashcard';

interface Exam { id?: number; name: string; code: string; }
interface Domain { id?: number; examId: number; name: string; weight: number; order: number; }
interface Topic { id?: number; domainId: number; parentTopicId?: number; name: string; description: string; order: number; status: TopicStatus; }
interface Resource { id?: number; topicId: number; resourceType: ResourceType; title: string; url?: string; notes?: string; createdAt: number; }
```

Status cycle: `not_started → in_progress → completed` (forward only, stops at completed).

## Testing Strategy

| Layer | What to Test | Approach |
|-------|-------------|----------|
| Unit | Progress calculations, status cycle logic, URL validation | Pure functions, no DOM |
| Unit | Seed data completeness (4 domains, correct weights) | Import seed, assert counts |
| Unit | Zustand store actions (toggle sidebar, set expanded set) | `act()` + assertions |
| Integration | StatusBadge: click → cycle → DOM updates | RTL + fake Dexie |
| Integration | ResourceForm: valid/invalid URL → submit enabled/blocked | RTL + mock Dexie |
| Integration | ProgressRing: calculates correct percentage from topic set | RTL + fake Dexie |
| Integration | Dexie schema: tables exist, seed inserts once | FakeIndexedDB or Dexie memory |

No E2E — the app is a single-user SPA with no backend to justify Playwright/Cypress for MVP.

## Migration / Rollout

No migration required — greenfield project. Future schema changes use `db.version(N).stores(...)` with upgrade handlers per the local-persistence spec.
